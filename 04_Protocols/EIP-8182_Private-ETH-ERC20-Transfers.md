## 概要

- **プロトコル名**: EIP-8182（Private ETH and ERC-20 Transfers）
- **開発主体**: Tom Lehman
- **対象ユースケース**: ネイティブプライバシーを持つ ETH・ステーブルコイン送金、企業間決済、給与支払い
- **対応チェーン / ネットワーク**: Ethereum メインネット（コア変更・ハードフォーク必要）
- **ライセンス**: ERC（オープン標準）

---

## 何を解決するプロトコルか

### 課題

既存のプライバシーアプローチはいずれも限界がある：

- **アプリレイヤーミキサー**（Tornado Cash）: 使用事実が追跡可能・規制リスク大
- **ERC-5564 ステルスアドレス**: 受取人プライバシーのみ・特殊アドレス形式が必要
- **EIP-7503 ZK Wormhole**: Stagnant・金額プライバシーなし
- **Aztec 等の ZK rollup**: 別の実行環境が必要、既存ウォレット・ENS と断絶

結果として、**普通のウォレットから普通のアドレスへ**プライベートに送金する手段が存在しない。

### アプローチ

プライベート送金を Ethereum プロトコルの「一等市民」として扱う：

- グローバルなシールドプールをプロトコルレイヤーで管理
- ZK 証明検証をプリコンパイルとして組み込み（ガスコスト最適化）
- 受取人は標準の Ethereum アドレス / ENS で指定可能
- **アトミックな「シールド → 操作 → 再シールド」フロー**を実現

## 技術仕様

### アーキテクチャ

- **システムコントラクト**を固定アドレスにデプロイ（EIP-4788 方式、管理者なし・アップグレード機構なし）
- 変更は Ethereum ハードフォークによってのみ可能
- 固定アドレスのコントラクトが以下のグローバル状態を管理：

```
┌─────────────────────────────────────┐
│       シールドプール状態             │
│  - Note-commitment ツリー           │
│  - Nullifier セット                 │
│  - ユーザー鍵レジストリ             │
│  - デリバリー鍵レジストリ           │
│  - 認可ポリシーレジストリ           │
└─────────────────────────────────────┘
```

### 主要コンポーネント

| コンポーネント | 説明 |
|-------------|------|
| Note-commitment ツリー | シールドプール内の資産残高をコミットメントとして管理するマークルツリー |
| Nullifier セット | ノートの二重使用を防ぐためのオンチェーン記録 |
| ユーザー鍵レジストリ | 一度登録すると既存ウォレットでプライベート受信が可能になるレジストリ |
| デリバリー鍵レジストリ | 受取人が自分の hidden owner identifier を登録する仕組み |
| 認可ポリシーレジストリ | 送金条件・コンプライアンスポリシーを ZK 証明ベースで定義 |
| ZK 証明プリコンパイル | EVM に組み込んだ証明検証プリコンパイル（低ガスコスト） |

### API / インターフェース

- ユーザーは**一度だけ**ユーザー鍵レジストリに登録 → 以降は通常の Ethereum アドレスで受信可能
- 送信者は受取人の標準アドレスまたは ENS を指定するだけでプライベート送金が成立
- アトミックフロー例：`ERC-20 資金をシールド → DEX でスワップ → 再シールド → 別ウォレットに出金`

### 対応通貨・アセット

- ETH（ネイティブ）
- ERC-20 全般（USDC・USDT・DAI・EURC 等のステーブルコインを含む）

| プロトコル | 違い |
|-----------|------|

| Tornado Cash | 廃止・制裁済み。使用事実が追跡可能 |

## 決済事業者・PSP としての評価

### 統合のしやすさ

- **採択・実装されれば**、既存の Ethereum ウォレットから標準アドレス宛てにプライベート送金が可能
- PSP 側の特別な実装は最小限で済む（受取人は一度レジストリ登録するだけ）
- ただし採択まで数年・実装後も普及に時間が必要

### コンプライアンス対応

- 認可ポリシーレジストリにより ZK ベースのコンプライアンス証明が設計上可能
  例：「送金者は KYC 済み」「送金先は制裁リストに含まれない」を ZK 証明として添付
- **実際の規制当局の受け入れは未知数**
- ファイナンシャルアクションタスクフォース（FATF）の Travel Rule への対応方法が明示されていない
- 開示鍵（viewing key）による選択的透明性を設計に組み込めるかどうかが実装次第

### スケーラビリティ・コスト

- ZK 証明プリコンパイル組み込みによりオンチェーン検証コストを大幅削減
- シールドプールのノートツリーが肥大化した場合のパフォーマンス維持が課題
- L1 コア変更のため L2 への影響は別途検討が必要

### リスク・懸念点

- **Draft 段階**：仕様が大幅に変更される可能性が高い
- コア変更（ハードフォーク）は政治的・技術的コンセンサスのハードルが極めて高い
- メンプール暗号化・ネットワーク層の匿名化は本 EIP のスコープ外であり、エンドツーエンドのプライバシーには追加対策が必要
- ウォレット側の UX 変更も別途必要

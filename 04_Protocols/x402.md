# x402

HTTP 402 Payment Required を実用化し、AI エージェントや機械が API・コンテンツ・デジタルサービスをリクエスト単位で支払えるようにする決済プロトコル。Coinbase が主導し、2026 年 4 月に Linux Foundation 傘下の x402 Foundation へ移管された。

---

## 何を解くプロトコルか

x402 が解く中心課題は、従来のカード決済やアカウント課金では扱いにくい **API 呼び出し単位・サブセント単位の機械決済**。

- エージェントが事前アカウントなしに有料 API を呼び出す
- AI 推論、データ取得、コンテンツ閲覧をリクエスト単位で課金する
- マーチャント / API 提供者が暗号資産ウォレットを直接運用せずに受け取る
- 大量の小額決済をバッチ化してオンチェーンコストを下げる

---

## レイヤー上の位置づけ

| 項目 | 内容 |
|---|---|
| 主体 | Coinbase / x402 Foundation |
| 主なレイヤー | 支払いプロトコル層 |
| 決済レール | 主にオンチェーン stablecoin / USDC |
| 強み | HTTP との親和性、API 課金、マイクロペイメント、OSS 標準化 |
| 弱み / 未解決 | 実需取引量、非 crypto 加盟店の受け入れ、認可・責任分担 |

x402 は「エージェントを信頼してよいか」を単独で解くものではない。AP2、Visa TAP、ERC-8004 などの認証・認可レイヤーと組み合わせる前提で見るのが自然。

---

## 他プロトコルとの関係

| 相手 | 関係 |
|---|---|
| AP2 | AP2 が認可・委任、x402 が支払い実行を担う補完関係 |
| MPP | API / machine payment 領域では競合。Stripe 加盟店網では MPP、オープンな HTTP / stablecoin 決済では x402 が強い |
| Visa TAP | TAP が trusted agent の識別、x402 がオンチェーン決済実行を担う可能性 |
| ERC-8183 | ERC-8183 がジョブ合意・エスクロー、x402 が支払い実行を担う補完関係 |

# A402

## Overview

A402 は、Web3 payment と Web2 service execution を Atomic Service Channels で結び、`service delivered if and only if payment settles` を狙う trust-minimized architecture prototype。

## User Goal

paid API や machine payments を公開したいが、paid-but-denied / unpaid-service のような atomicity failure を避けたい。

## Business Goal

x402 より強い fulfillment binding を持つ payment architecture を作り、merchant 側の trust cost と revenue leakage を減らすこと。

## Category

x402, agent-payments

## Core Workflow

1. payer と service provider が Atomic Service Channel を開く。
2. service execution と payment condition を同じ channel state に結びつける。
3. TEE-assisted adaptor signatures を使い、service result の delivery と payment finalization を連動させる。
4. settlement は aggregated に on-chain 反映し、high-frequency micropayments の overhead を下げる。

## Pricing / Business Model

research artifact。商用化ポイントは atomic paid API gateway、verified service channel infra、high-frequency machine billing rail にある。

## Differentiation

x402 のような HTTP-native payment をそのまま簡略化するのではなく、`service execution binding` を payment primitive にまで引き下げている点が核心。

## Operational Notes

- TEE 依存は merchant adoption の大きな friction になり得る。
- channel lifecycle 管理と liquidity management が self-host 運用を難しくする可能性がある。
- merchant が必要とする assurance と導入複雑性の trade-off を見極める必要がある。

## Risks / Open Questions

- public implementation や benchmark harness の継続公開状況は未確認。
- API seller が x402 から移行するほどの operational gain があるか不明。
- dispute や partial fulfillment をどう扱うかが source だけでは見えにくい。

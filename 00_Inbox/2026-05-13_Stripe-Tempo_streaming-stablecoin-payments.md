## 概要

- Stripe は Stripe Sessions 2026 で、AI ビジネス向けの新決済機能として、Tempo 上のステーブルコインを活用するストリーミング決済を発表した。
- NADA NEWS によると、ストライプジャパンが 2026-05-13 に国内向けに発表内容を紹介した。
- 仕組みは、Stripe 傘下の Metronome による利用状況トラッキングと、Tempo 上のステーブルコインによるマイクロペイメントを組み合わせるもの。

## 何が起きたか

- AI モデルの利用量単位であるトークン消費に応じて、リアルタイムに課金・回収する決済機能が紹介された。
- 従来の決済システムでは、ミリ秒単位で発生する極少額支払いを処理しづらく、AI サービス提供者が利用料回収前に推論コストなどのインフラ費用を先に負担しやすい。
- 今回のストリーミング決済では、AI トークンが使用されたタイミングで、企業がトークンごとの支払いを受け取れるようにする構想。
- 日本での提供時期や提供条件は未公表。

## 確認された事実

- 発表イベントは Stripe Sessions 2026。
- 関連コンポーネントは Metronome、Tempo、ステーブルコイン・マイクロペイメント。
- 対象ユースケースは、AI プロダクトにおける短時間・大量のトークン消費、AI エージェントによる少額・高頻度決済。
- Tempo は Stripe と Paradigm が立ち上げた決済特化型 L1 と説明されている。
- 同じ発表群には、Link を活用した AI エージェント向けウォレット、ステーブルコイン決済拡大、グローバル送金、デジタル資産アカウントも含まれる。

## なぜ重要か

### 決済事業者視点

- Stripe が既存のカード決済・チェックアウト中心の課金から、AI 推論や API 利用に近い「連続的な利用量課金」へ決済対象を拡張している。
- 決済処理の単位が「注文」から「トークン消費」「API コール」「セッション内イベント」へ細かくなると、PSP には低コスト・高頻度・即時性のある決済基盤が必要になる。

### 加盟店視点

- AI サービス事業者にとって、推論コストを先に負担して後から月次請求する構造を緩和できる可能性がある。
- ただし日本提供時期・条件が未定のため、国内事業者がすぐ実装できる機能としては扱えない。

### プロダクト視点

- Metronome が利用量計測、Tempo が少額・高頻度の価値移転、Stripe が請求・精算・ウォレット・コンプライアンスを担う構成に見える。
- MPP が「エージェントが支払う」プロトコルだとすると、この機能は「AI サービス利用に対して連続的に回収する」課金レイヤーに近い。
- x402 の 1 リクエスト単位課金と比較すると、Stripe は Metronome と組み合わせたセッション/利用量ベースの商用課金に強みを置く可能性がある。

### 規制 / リスク視点

- ステーブルコインを使う場合、発行体、準備資産、送金規制、本人確認、返金・取消、消費者保護の責任分担が論点になる。
- AI トークン単位のリアルタイム課金は、利用者への価格表示、予算上限、異常利用時の停止、誤課金時の返金設計が不可欠。

## 我々への示唆

- いま検討すべきこと:
  - AI / API / agentic commerce の課金設計では、月次請求・前払いクレジット・都度決済だけでなく、ストリーミング回収モデルも選択肢に入れる。
  - x402 と MPP / Tempo の比較軸に「課金粒度」「利用量計測との接続」「精算単位」「法定通貨圏との接続」を追加する。
- 後で深掘りすべきこと:
  - Metronome と Stripe Billing / Payments / Bridge / Tempo の責任分界。
  - ストリーミング決済の決済確定タイミング、失敗時の再試行、返金、支出上限、加盟店精算通貨。
  - 日本での提供可能性と、資金決済法・暗号資産交換業・電子決済手段関連規制との関係。
- 今は優先度が低いこと:
  - 日本での即時実装検討。提供条件が未公表のため、現時点では構想・競争軸として監視する。

## ありそうな示唆

- Stripe は AI エージェント決済を「購入代行」だけでなく、「AI インフラ利用料の回収」という B2B / developer economy 側の課題に広げている。
- Tempo は単なるステーブルコイン送金チェーンではなく、Stripe のメータリング、請求、ウォレット、エージェント認可と組み合わさる決済実行レイヤーとして設計されている可能性が高い。

## 推測 / 監視ポイント

- MPP、Link エージェントウォレット、Metronome、Tempo の関係が公式ドキュメントでどう整理されるか。
- ストリーミング決済が x402 のようなオープン API 課金標準と競合するのか、Stripe エコシステム内の実装パターンに留まるのか。
- Visa が Tempo のアンカーバリデーターとして関与する場合、カード決済ネットワークとステーブルコイン・マイクロペイメントがどう接続されるか。

## 未解決の論点

- Stripe / Tempo / Metronome の一次情報で、ストリーミング決済の正式名称、API、対応地域、提供時期を確認できていない。
- 日本での提供可否、利用条件、規制上の取り扱いは未公表。
- 利用者側の支払い原資がステーブルコインなのか、カード/銀行/Stripe 残高から抽象化されるのかは不明。
- マイクロペイメントの手数料設計、最低課金単位、決済失敗時の UX は未確認。

    x402 Sign-In-With-X（SIWX）が Go 言語に対応——クロス SDK エージェント認証が完成

    概要¶

    x402 プロトコルの認証拡張である Sign-In-With-X（SIWX）が Go 言語に完全対応した（PR #2485、2026-06-19 マージ）。SIWX は x402 の「都度課金」に加えて「署名チャレンジによる認証」をサポートする拡張機能で、エージェントが一度認証すれば繰り返しアクセス時に毎回 USDC を支払わずに済む仕組みを提供する。

    何が起きたか¶

    • PR #2485「feat(go): add sign-in-with-x server and client support」が 2026-06-19 にメインブランチへマージ
    • Go の x402 SDK にサーバー（保護エンドポイント側）とクライアント（アクセス側）の双方の SIWX 実装が追加された
    • 認証フロー：サーバーが SIWX チャレンジを発行 → クライアント（エージェント）が EIP-191 署名で応答 → サーバーが EIP-1271 スマートウォレット検証を含む署名検証を実施 → 認証成功でリソースを返す（支払い不要）
    • クロス SDK テスト結果（Base Sepolia での検証）：
    • TypeScript クライアント → Go サーバー：auth-only ルート・有料ルートの双方で成功
    • Go クライアント → TypeScript サーバー：同上で双方成功
    • 4件のオンチェーン決済トランザクションで正常な決済フロー＋SIWX 認証の連続動作を確認
    • EIP-1271（スマートウォレットの署名検証標準）に対応しており、EOA だけでなくスマートウォレット（Coinbase スマートウォレット等）でも SIWX が使える

    確認された事実¶

    • GitHub x402-foundation/x402 PR #2485 が 2026-06-19 にメインブランチへマージ（一次情報）
    • Go の x402 SDK に SIWX サーバー拡張と SIWX クライアントサポートが追加
    • Base Sepolia (eip155:84532) でのクロス SDK 互換テストが完了・成功
    • EIP-191（個人メッセージ署名）と EIP-1271（スマートウォレット署名）の双方をサポート
    • 同日（2026-06-19）にバージョンバンプ関連コミット（chore: version typescript packages、chore(go): release）が複数投入され、正式リリースとして整備された

    なぜ重要か¶

    決済事業者視点¶

    • Go は高トラフィックのバックエンドサービス（金融システム・API ゲートウェイ等）で広く使われる言語。Go 対応により、既存の Go ベースの API インフラに x402 SIWX 認証を組み込む企業が増えうる
    • SIWX は「一度認証したエージェントには都度課金しない」モデルを可能にする。PSP がサブスクリプション型 x402 課金モデルを設計する際の参照実装として活用できる

    加盟店視点¶

    • 認証型アクセスは API キーの代替として機能する。エージェントが SIWX で認証済みであれば、同一セッション中のリソースアクセスに毎回 USDC が不要になる——フリクションが大幅に減る
    • 「繰り返し小額支払い vs. 一回認証後アクセス」の柔軟な設計が可能になり、加盟店が顧客（エージェント）の利用パターンに合わせた価格設定ができる

    プロダクト視点¶

    • x402 のクロス SDK 対応（TypeScript・Go・Python）が SIWX 認証レイヤーでも揃いつつある。多言語のバックエンドが混在する企業システムで x402 を標準採用できる基盤が整ってきた
    • SIWX（Sign-In-With-X）は SIWE（Sign-In-With-Ethereum）の拡張であり、既存の Web3 認証インフラ（WalletConnect・Rainbow Kit 等）との統合パスが見えやすい

    規制 / リスク視点¶

    • SIWX によるセッション型認証が浸透すると「エージェントは誰の代わりに認証しているか」という問いが規制上重要になる。エージェントが本人認証なしに API にアクセスできる経路が増えれば、AML・本人確認（KYC）ギャップが拡大しうる
    • ただし EIP-1271 スマートウォレット検証を使えば、on-chain のスマートコントラクト経由でエンティティの同一性確認が可能という点で、規制対応の技術的手段もある

    我々への示唆¶

    • いま検討すべきこと:
    • 自社 Go バックエンドでの x402 SIWX 実装可能性の技術評価（PR #2485 の実装を参照）
    • 「都度課金型（每リクエスト USDC）」と「認証型（SIWX 一度認証後、一定期間フリー）」のハイブリッド価格モデルの設計を検討
    • 後で深掘りすべきこと:
    • SIWX セッションの有効期限管理・再認証フローの標準化（現時点では仕様が成熟途上）
    • EIP-1271 スマートウォレット（Coinbase SW・Safe 等）と SIWX の組み合わせによるエージェント身元保証の可能性
    • 今は優先度が低いこと:
    • Kotlin / Java / Swift / Rust への SIWX 対応移植（Go・TS・Python の主要言語が揃った後の対応）

    ありそうな示唆¶

    • x402 SIWX の Go 対応により「TypeScript フロントエンド + Go バックエンド」という企業システムの標準スタックで x402 が完全動作する環境が整った。今後はエンタープライズ向け x402 採用ガイドが登場し、実際の企業ユースケース（内部 API アクセス・B2B マイクロサービス間決済）での採用が進むと見られる

    推測 / 監視ポイント¶

    • Python SIWX 実装との三言語クロス SDK 互換テスト完了の告知
    • SIWX を使ったサブスクリプション型 x402 課金モデルの公式ドキュメント化
    • EIP-1271 検証を活用した企業向け「エージェント身元証明」ソリューションの登場

    未解決の論点¶

    • SIWX セッションの有効期限管理が標準化されているか（各実装が独自に定義している可能性がある）
    • SIWX 認証に使うウォレットアドレスが「エージェントのアドレス」か「エージェントを操作するユーザーのアドレス」かの設計上の曖昧さ
    • SIWX セッション中に同一エージェントが別エンドポイントに横断アクセスできるか（セッションのスコープ設計）

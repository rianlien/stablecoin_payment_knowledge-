# Skill Improver — Nightly Self-Improvement Loop

> **層: Mechanism（OS層）。** Contract / Intent（CLAUDE.md）と他の Mechanism（skills / prompts）の整合を保つ手段。改善 PR でも非対称ルールを守る（Mechanism の都合で上流を黙って書き換えない）。

## Purpose
夜間に自律実行し、このリポジトリのスキルファイルとプロンプトを自動改善する。
ユーザーが操作中に気づいた問題や、ノートの品質チェックで発見したルールの曖昧さを、
翌朝のPRとして提案する。

> このスキルは unattended 実行専用。ユーザーとのインタラクティブなセッションでは実行しない。

---

## Phase 1: Signal Collection

以下の3ソースからシグナルを収集する。

### 1a. Git history analysis
```bash
git log --since=14.days --oneline --no-merges
git log --since=14.days --pretty="%s%n%b" --no-merges -- CLAUDE.md .claude/skills/ .claude/prompts/
```
- 「fix」「修正」「追記」「ルール」「clarify」を含むコミットメッセージを優先確認
- CLAUDE.md やスキルファイルへの変更コミットをリストアップ
- 繰り返し修正されている箇所を特定する

### 1b. Note compliance check
過去14日間の `/00_Inbox/` ノートに対して以下を確認する。

必須 frontmatter フィールドの有無:
- `article_published_date`、`underlying_event_date`、`primary_source_date`
- `importance`（High / Medium / Low）
- `confidence`（High / Medium / Low）
- `tags`（配列形式）
- `entity`、`category`、`primary_category`

その他の確認事項:
- ファイル名形式の遵守（`YYYY-MM-DD_<entity>_<short-slug>.md`）
- 推奨タグ外のタグ使用パターン
- `source_url` の有無

### 1c. Daily Brief quality check
過去14日間の `/01_Daily-Briefs/` を確認:
- リンク形式の逸脱（`[Source Name（YYYY-MM-DD）](URL)` 形式が守られているか）
- 「該当なし」の記録漏れ
- テンプレートの構造からの逸脱

### 1d. 思想整合チェック（Intent → Contract → Mechanism）
設計思想は `_meta/PHILOSOPHY.md`、配線図は `_meta/ARCHITECTURE.md` を参照。以下のドリフトを検出する:
- Mechanism（`.claude/{skills,prompts,hooks}` / `mkdocs.yml` / CI）固有の都合が、CLAUDE.md の Intent / Contract 節に無印で直書きされていないか（昇格漏れ・委譲漏れ）
- skill / prompt が参照する CLAUDE.md のセクション名が、実際の見出しと食い違っていないか
- 各 `index.md` の「層宣言」が実態とドリフトしていないか

## Phase 2: Analysis

収集したシグナルから改善ポイントを特定する。以下の優先順位で評価する:

1. **High priority**: 複数のノートで同じ違反が確認できる → ルールが不明確
2. **Medium priority**: git history に修正パターンが見られる → ルール補完が必要
3. **Low priority**: 単発の逸脱 → スキップ（単発はルールの問題でない可能性大）

改善対象外のシグナル:
- コンテンツの質（事実の正確さ、要約の深さ）— 人間が判断すべき
- 単発の判断ミス — ルール化しても過剰適合になる
- CLAUDE.md の核心的な運用方針（paths, date conventions, output destinations）

---

## Phase 3: Improvement Proposal

改善点が特定できた場合のみ、以下を実行する。

改善対象ファイル（候補）:
1. `CLAUDE.md` — メインルールセット（Intent ＋ 横断 Contract）
2. `.claude/rules/*.md` — path 固有 Contract（inbox-notes / daily-briefs / concept-pages）
3. `.claude/skills/daily-news/SKILL.md` — デイリーニュース収集スキル（手順）
4. `.claude/prompts/daily_routine_prompt.txt` — 日次ルーティンプロンプト

変更の制約:
- 1回の実行で変更するファイルは最大3ファイル
- 1ファイルへの変更は最小限（diff が読みやすい形を維持）
- 既存のセクション構造を壊さない
- 削除より追記・明確化を優先する

---

## Phase 4: Quality Gates

変更を提案する前に、以下を順番に確認する（各最大3回まで反復）:

### Gate 1: Verify（効果確認）
「この変更は、検出したシグナルを実際に防げるか?」
- 検出した違反ケースに変更後のルールを適用し、問題が解決するか確認する
- 解決しない場合は変更内容を見直す

### Gate 2: Review（一貫性確認）
「変更後もスキルファイル全体が矛盾なく一貫しているか?」
- 変更した箇所と既存ルールの矛盾がないか確認する
- CLAUDE.md の既存ルールと衝突していないか確認する

### Gate 3: Security（安全性確認）
「変更に個人情報・認証情報が含まれていないか?」
- ユーザーメールアドレス、API key、内部URL が含まれていないことを確認する

### Gate 4: Asymmetry（非対称ルール検査）
「この変更は Mechanism の都合で Contract / Intent を黙って書き換えていないか?」
- 変更が Mechanism 起因なら、対応する Contract / Intent（CLAUDE.md / `_meta/`）を先に更新してから Mechanism を直しているか
- スクリプト固有の理由を CLAUDE.md（Contract）に直書きしていないか。していれば抽象化して Contract に昇格し、理由は `_meta/ARCHITECTURE.md` 等の Mechanism 側へ移す
- **Intent（収集の目的・論点の枠組み）は人間の領域。** 自動改善で書き換えず、必要なら PR 本文で「Intent への変更提案」として明示するに留める

---

## Phase 5: Output

### 改善点がある場合
feature branch を作成し、変更を適用してから PR を作成する。

ブランチ名: `claude/skill-improve-YYYY-MM-DD`
PR タイトル: `chore(skills): nightly skill auto-improvement YYYY-MM-DD`

PR body には以下を記載する:
- 検出したシグナル（何のパターンに基づくか）
- 変更内容の要約（何をどう変えたか、なぜそれが改善か）
- 変更前後の diff サマリー
- `🤖 Generated by skill-improver (unattended run)`

### 改善点がない場合
何も変更せずに終了する。空PR は作らない。

実行後は以下を出力する:
```

[skill-improver] 変更対象ファイル: <ファイル名リスト>
[skill-improver] Quality gates: Verify ✓ / Review ✓ / Security ✓ / Asymmetry ✓
[skill-improver] PR作成: <PR URL> または「改善点なし、終了」
```

# 層: Mechanism（OS層）。CLAUDE.md の Contract（概念ページ・来歴）を満たす実行プロンプト。ルール変更は先に CLAUDE.md / _meta/ を更新する（非対称ルール）。

あなたは stablecoin / agentic commerce リサーチナレッジベースの概念ページ作成担当エージェントです。
このリポジトリの CLAUDE.md を必ず読み込み、そこに記載された全ての運用ルールに従ってください。

## 実行情報
- TODAY: ${TODAY}
- REPO_ROOT: ${REPO_ROOT}（このリポジトリのルート。routine は REPO_ROOT を cwd として実行される）

## ミッション
`auto-todo` ラベルの GitHub Issue を処理し、不足している概念ページを新規作成する。
web 検索は行わない。既存ノートの内容のみを根拠にページを作成する。

---

## タスク 1: 処理対象 Issue の取得

```bash
gh issue list --label auto-todo --state open --json number,title,body --limit 20
```

Issue は4種類ある:
- `missing_company_page`: `/06_Entities/Companies/` に企業ページが存在しない
- `missing_organization_page`: `/06_Entities/Organizations/` に組織ページが存在しない
- `missing_protocol_page`: `/04_Protocols/` にプロトコルページが存在しない
- `missing_topic_page`: `/03_Topics/` にトピックページが存在しない

`missing_company_page` ラベルでも、対象エンティティが政府機関・規制機関・標準化団体・業界コンソーシアムの場合は `/06_Entities/Organizations/` に作成する。

処理対象がない場合（0件）はそのまま終了する。

---

## タスク 2: 各 Issue の実装

Issue ごとに以下を実行する。1回の実行で処理する Issue は最大 5 件。

### 2-1. 参照ノートの読み込み
Issue body に記載されている参照ノート（`/00_Inbox/` 内）を全て読む。
ページの内容はこれらのノートにある事実のみから作成する。推測は推測と明記する。

### 2-2. 既存ページをフォーマット参照として読む
同じ種類の既存ページを 1〜2 件読み、フォーマット・文体・構成を確認する。

**企業ページの参照例**: `/06_Entities/Companies/Circle.md` または `Coinbase.md`
フォーマット:
```
---
title: "<企業名>"
type: company
updated: TODAY
category: <payments|infrastructure|regulation など>
tags: [...]
---

# <企業名>

<企業の1〜2文の位置づけ>

---

## 現状の要約

### 何をしているか
- <事実ベースの箇条書き>

### 見立て
<この vault での追い方と不確実性>

---

## Inbox との紐付け

| 日付 | ノート | 位置づけ一行 |
|---|---|---|
| YYYY-MM-DD | [[ノート名]] | 一行要約 |
```

**プロトコルページの参照例**: `/04_Protocols/ERC-8183_Agentic-Commerce.md`
フォーマット:
```

```

### 2-3. ページの作成

以下のルールに従って新規ファイルを作成する:

**ファイル名**:
- 企業（民間）: `/06_Entities/Companies/<EntityName>.md`（英語表記、スペースはハイフン）
- 組織（政府機関・規制機関・標準化団体・業界コンソーシアム）: `/06_Entities/Organizations/<EntityName>.md`
- プロトコル: `/04_Protocols/<IDENTIFIER>_<Short-Name>.md`
- トピック: `/03_Topics/<slug>.md`（kebab-case）

**Companies vs Organizations の判断基準**:
- Companies: 民間企業（fintech、startup、上場企業、VC 投資先など）
- Organizations: 政府機関（Fed、FDIC、FinCEN、NCUA、OCC、SEC、CFTC など）、標準化団体（Linux Foundation、FIDO Alliance、ISO など）、業界コンソーシアム（x402 Foundation、Mojaloop など）、非営利財団

**内容の制約**:
- 事実（参照ノートに記載されている内容）と推測（そこから導かれる解釈）を分ける
- 推測には「〜と見られる」「〜の可能性がある」などを付ける
- 参照ノートの `underlying_event_date` より新しい情報を作らない
- `updated` は TODAY を使う

### 2-4. 既存概念ページへの逆リンク追加（任意）
新規ページが関連する既存の概念ページにリンクを追記できる場合は追記する。
対応する概念ページが存在しない場合はスキップ。

## タスク 3: PR 作成

1 件以上のページを作成した場合のみ実行する。

```bash
git checkout -b automated/auto-todo-$TODAY
git add 06_Entities/Companies/ 06_Entities/Organizations/ 04_Protocols/ 03_Topics/ 05_MOCs/
git diff --cached --quiet || git commit -m "feat(pages): auto-create concept pages from auto-todo issues $TODAY"
git push origin automated/auto-todo-$TODAY
gh pr create \
  --title "feat(pages): concept pages from auto-todo ($TODAY)" \
  --body "## 作成した概念ページ

{ページ一覧（ファイル名・対応 Issue 番号）}

## 根拠
各ページは `/00_Inbox/` の参照ノートのみを根拠に作成。web 検索は行っていない。

## 確認ポイント
- [ ] 事実 / 推測の区別が適切か
- [ ] フォーマットが既存ページと一致しているか
- [ ] 参照ノートへのリンクに誤りがないか

🤖 Generated by auto-todo-executor (unattended run)" \
  --base main
```

## タスク 4: Issue へのコメント

PR を作成した場合、処理した各 Issue に PR URL をコメントする。

```bash
gh issue comment <issue_number> --body "PR を作成しました: <PR_URL>
作成したページ: `<ファイル名>`
内容を確認後、Issue をクローズしてください。"
```

Issue 自体はクローズしない（PR レビュー後に人間がクローズする）。
